Verifiable Execution for AI Systems

A field note on inference, agents, and evidence.

research.tribunus.dev
Open systems research for inspectable AI infrastructure.

The execution layer is a safety boundary.

Most AI systems are evaluated at the output layer: did the model answer correctly, did the agent complete the task, did the benchmark improve.

Tribunus focuses on what happened beneath that output.

  • What model ran.
  • Which backend executed it.
  • Where tensors and state lived.
  • What data crossed a boundary.
  • What authority the agent exercised.
  • What state changed.
  • What evidence remains after execution.

“Outputs alone cannot explain whether an AI system acted within its authority, executed correctly, or can be reproduced.”

Research Question

How can AI systems emit enough structured evidence that a result can be inspected, replayed, compared, or rejected?

Opaque execution becomes a systems risk.

Hidden materialization

A tensor may appear local, zero-copy, or persistent while a backend silently copies, transforms, or relocates it.

Unscoped agent authority

An agent may appear to perform one user request while actually invoking tools with broader mutation authority than the user intended.

Unverifiable backend behavior

A successful output does not by itself establish which kernel, model, quantization path, fallback route, or hardware backend produced it.

Principles:
  • No hidden copies.
  • No implicit authority.
  • No unsupported correctness claims.
  • No evidence-free execution.

From model graph to execution receipt.

The portable layer is the compiler and scheduler semantics. The backend-specific layer is lowering, memory model, artifact format, launch mechanism, ABI, and completion integration.

Portability does not mean pretending hardware is the same.

Target Memory reality Execution reality What Tribunus must make explicit
Apple Metal / MLX Unified memory and platform-specific sharing paths Metal command encoding and GPU-resident resources Explicit layout, materialization, artifact, and completion behavior
AMD ROCm / HIP Discrete VRAM with explicit host-device transfer and residency HIP streams, events, kernels, ROCm libraries Upload, pinning, stream dependencies, VRAM lifetime
Intel XPU / SYCL / Level Zero Integrated or discrete memory depending on platform SYCL queues, USM, Level Zero allocation/event semantics Allocation form, queue/event behavior, locality assumptions
Tenstorrent Accelerator-local memory and explicit data movement TT-Metal or TT-NN execution topology Placement, circular buffers, device-resident weights, mesh topology

The universal promise is not zero-copy everywhere.

“No hidden copies. Explicit materialization whenever required. Target-specific zero-copy only where the backend can prove it.”

What Tribunus is building and validating.

  • Implemented
    • Capability-scoped runtime primitives.
    • Durable execution receipts and coordination patterns.
    • Backend-aware inference abstractions.
  • Experimentally validated
    • Compute-image qualification and numerical oracle workflows.
    • MLX / Metal execution paths and backend evidence collection.
  • Partially validated
    • Core ML qualification cells and numerical divergence classification.
    • Cross-backend placement and artifact-admission semantics.
  • Designed / in progress
    • Portable PhaseIR and ComputeImage compiler.
    • Heterogeneous Apple, AMD, Intel, CUDA, and accelerator lowering.
    • Federated mutual-aid inference and receipt verification.

An early ComputeImage validation artifact covers a Qwen2.5 0.5B workload, NF4 quantization, MLX Metal as the primary route, Accelerate fallback, and oracle-based validation. See the evidence registry for scope and limitations.

EVIDENCE STATUS: EARLY / EXPERIMENTAL

Research questions worth solving.

  1. Numerical validation across hardware.

    How should quantized models and fused kernels be admitted when numerical behavior differs across GPU architectures, drivers, libraries, and fallback paths?

  2. Privacy-preserving evidence.

    How can execution receipts remain useful for audit and reproduction without exposing private project state, user data, or proprietary model artifacts?

  3. Semi-trusted federated execution.

    How can peers contribute inference capacity while making resource claims, execution receipts, and failure modes inspectable?

  4. Agent authority at the OS boundary.

    How should local AI systems constrain filesystem, shell, credential, browser, and plugin access so capability scopes remain meaningful outside the model runtime?

Tribunus is looking for collaborators who can reproduce evidence, inspect backend contracts, validate numerical methods, test real hardware, or challenge the architecture.

Relevant collaborators:

ML compiler engineers. GPU/runtime engineers. AI safety researchers. Distributed-systems researchers. Hardware vendors and accelerator teams. Open-source contributors.
